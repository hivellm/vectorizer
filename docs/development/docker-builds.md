# Docker build pipeline

This document is the operator runbook for building and publishing the
official `hivehub/vectorizer` Docker image. It supersedes the ad-hoc
build instructions that used to live in `Dockerfile` header comments
and brings them in line with the dual-registry pipeline shipped in
phase10 (`.rulebook/tasks/phase10_optimize-docker-build-time/`).

## TL;DR

| Audience | Command |
|----------|---------|
| Maintainer cutting a release | `.\scripts\docker\build-push.ps1 -Tag 3.x.y` |
| Local-only build (no push) | `.\scripts\docker\build.ps1 -Tag dev` |
| Re-push an already-built tag | `.\scripts\docker\push.ps1 -Tag 3.x.y` |
| CI release publish | tag `vX.Y.Z` triggers `.github/workflows/release-artifacts.yml` |

All three local scripts default to the `hivehub/vectorizer-cache:buildx`
registry cache. Pass `-NoCache` to force a cold build.

## Why this pipeline exists

Cold local Docker builds of `hivehub/vectorizer:3.2.0` measured
**30–45 minutes** on a 33 GB Docker Desktop allocation (Windows host,
multi-arch `linux/amd64,linux/arm64`). The breakdown (captured during
the v3.2.0 push session):

| Stage | Time |
|-------|------|
| `cargo chef cook` (deps warm) | 10m 23s |
| Workspace compile (per arch) | 3–4m amd64 / 5–6m arm64 |
| `cargo install cargo-sbom` + `cargo sbom` (per arch) | ~2m × N arches |
| arm64 anything via QEMU emulation | 2–3× the native amd64 cost |
| BuildKit syft SBOM attestation | ~5s (already optimal) |

The cumulative pain compounded on every release cut and every Scout /
CVE re-bake. Phase10 reduces the cold path by removing the
`cargo-sbom` step (BuildKit syft attestation is the source of truth
already), drops LTO from the container build via a dedicated Cargo
profile, adds a buildx registry cache for layer reuse across builds,
and switches CI to native arm64 runners so QEMU is out of the loop.

Expected wall-clock after phase10:

- Cold local build: 15–20m (was 30–45m).
- Warm local build (single dashboard line edit): under 10m.
- CI release publish: each arch matrix job finishes in 5–8m on its
  native runner.

## Build artifacts

The pipeline produces:

- A multi-arch (`linux/amd64`, `linux/arm64`) manifest list pushed to
  both `ghcr.io/hivellm/vectorizer` (always, via `GITHUB_TOKEN`) and
  `hivehub/vectorizer` (when `DOCKERHUB_USERNAME` + `DOCKERHUB_TOKEN`
  repo secrets are set).
- Per-arch SBOM attestations generated by BuildKit's syft integration
  (`--sbom=true`); inspectable via
  `docker buildx imagetools inspect <image>`.
- Per-arch `mode=max` SLSA provenance attestations (`--provenance`).

The runtime image no longer carries `/vectorizer/vectorizer.spdx.json`
— Scout policy reads SBOM from the BuildKit attestation, not from a
file inside the image. See
`.rulebook/tasks/phase10_optimize-docker-build-time/specs/build/spec.md`
for the full requirements.

## Local build flow

### Multi-arch build + push (release path)

```powershell
.\scripts\docker\build-push.ps1 -Tag 3.2.1
```

This:
1. Spins up the `vectorizer-builder` buildx instance if it doesn't
   exist (docker-container driver, `linux/amd64,linux/arm64`).
2. Runs `docker buildx build --platform linux/amd64,linux/arm64`
   against `Dockerfile`, with `--cache-from` + `--cache-to type=registry,ref=hivehub/vectorizer-cache:buildx,mode=max`.
3. Generates SBOM attestation (`--sbom=true`) and SLSA provenance
   attestation (`--provenance=mode=max`).
4. Pushes the manifest list + per-arch images + cache layer to Docker
   Hub in one step.

**Prerequisites:** `docker login` against `docker.io` and `dhi.io`
(the latter is required for the `dhi.io/debian-base:trixie` runtime
base image pull).

### Local-only build (no push)

```powershell
.\scripts\docker\build.ps1 -Tag dev
```

Builds the host-arch image, materializes it into the local Docker
daemon (via `--load`), and reads the registry cache to skip
unchanged layers. Does not push anything anywhere.

### Cold build (no cache)

Append `-NoCache` to either script:

```powershell
.\scripts\docker\build-push.ps1 -Tag bench-baseline -NoCache
.\scripts\docker\build.ps1 -Tag bench-baseline -NoCache
```

Use this when measuring the baseline (e.g., to validate that the
cache is actually delivering the expected speedup).

## Cache lifecycle

The `hivehub/vectorizer-cache` repository on Docker Hub is dedicated
to buildx layer caching. Storage cost is ~3–5 GB; well within the
free-tier quota.

- **Tag:** only `:buildx` is used — the cache is overwritten in place
  on every push. There's no per-version cache; this is intentional
  (the cache is a layer pool, not a versioned artifact).
- **Invalidation:** changes to `Cargo.lock`, workspace `Cargo.toml`,
  or any file under `crates/` invalidate the `cargo chef cook` layer
  and force a re-warm. Dashboard edits, Dockerfile env tweaks, and
  documentation changes hit the cache cleanly.
- **Reset:** to drop the cache and force every consumer to rebuild
  from scratch, delete the `hivehub/vectorizer-cache:buildx` tag on
  Docker Hub. The next `build-push.ps1` run will re-seed it.

## Dual Cargo profile contract

The image binary is built with a dedicated Cargo profile,
`release-docker`, defined in workspace `Cargo.toml`:

```toml
[profile.release-docker]
inherits = "release"
lto = false        # workspace `release` uses lto = "thin"
codegen-units = 16 # workspace `release` uses 4
incremental = false
```

Trade-off: the shipped binary is ~10–15% slower on hot paths versus
the workspace `release` profile, in exchange for ~30% faster compile
and ~50% lower peak rustc memory. This trade is acceptable inside a
container (operators chasing peak throughput already build from
source with `cargo build --release` on the host).

The Dockerfile selects this profile via `ARG PROFILE=release-docker`
(default). Override at build time with
`--build-arg PROFILE=release` if you need the workspace profile in
the image (e.g., for benchmark parity).

The host-side `cargo build --release` flow is unchanged — the
`release` profile keeps `lto = "thin"` + `codegen-units = 4`.

## CI release publish flow

`.github/workflows/release-artifacts.yml::publish-docker` is a matrix
job that builds each arch on a native runner:

| Matrix entry | Runner | Platform |
|--------------|--------|----------|
| `amd64` | `ubuntu-latest` | `linux/amd64` |
| `arm64` | `ubuntu-24.04-arm` | `linux/arm64` |

Each matrix job:
1. Pulls the dashboard artifact (`dashboard/dist`, built once in
   `build-dashboard`) and the matching pre-built Linux binary
   (`binaries/linux-${arch}/vectorizer`, built natively in
   `build-linux-gnu-for-docker`).
2. Builds against `docker/Dockerfile.artifacts` (which only does
   `COPY` + `chmod` + `chown` — no Rust compile inside Docker).
3. Pushes by digest to `ghcr.io/hivellm/vectorizer` (no tag yet).
4. Uploads the digest as a tiny artifact for the manifest job.

The downstream `publish-docker-manifest` job:
1. Downloads both digests.
2. Logs into ghcr.io (always) and Docker Hub (when secrets exist).
3. Runs `docker buildx imagetools create` to assemble the multi-arch
   manifest list with the human-readable tags
   (`{{version}}`, `{{major}}.{{minor}}`, and `latest` on the default
   branch). The same per-arch digests are stitched into both the
   ghcr.io and Docker Hub manifest lists in one step each.
4. Inspects the published manifest to verify both arches and both
   attestations (SBOM + provenance) are present.
5. Runs `docker scout policy --org hivehub --platform linux/amd64`
   as a supply-chain gate; fails the workflow on policy regression.

### Required repo secrets

| Secret | Purpose | Optional? |
|--------|---------|-----------|
| `GITHUB_TOKEN` | ghcr.io push | Always present (auto-provisioned) |
| `DOCKERHUB_USERNAME` | Docker Hub push + Scout | Yes — gated via `if: env.DOCKERHUB_USERNAME != ''` |
| `DOCKERHUB_TOKEN` | Same | Same |

Without the Docker Hub secrets the workflow still publishes to
ghcr.io but skips the Hub push and the Scout gate.

## Troubleshooting

### Docker Desktop OOMs partway through `cargo chef cook`

Symptom: BuildKit container killed with `gRPC EOF` and
`cannot allocate memory` SIGKILL. Two earlier v3.2.0 attempts hit
this at an 8 GB Docker Desktop allocation.

Root cause: peak rustc memory of 4–6 GB on the `vectorizer` lib
monomorphization, multiplied by `CARGO_BUILD_JOBS` parallelism.

Fix: bump Docker Desktop memory to **at least 16 GB** (Settings →
Resources → Memory). 33 GB is comfortable. The Dockerfile already
caps `CARGO_BUILD_JOBS=2` to keep peak memory predictable; lowering
it further (`--build-arg CARGO_BUILD_JOBS=1`) is the last resort
before bumping the memory floor.

### `OPENSSL_DIR does not exist` from openssl-sys

Symptom: build fails inside a `RUN xx-cargo` step with
`OpenSSL include directory does not exist /Program Files/OpenSSL-Win64`.

Root cause: Docker Desktop for Windows + buildx desktop-linux leaks
the host `OPENSSL_DIR` env var into Linux RUN steps.

Fix: every affected `RUN` already prefixes with
`unset OPENSSL_DIR OPENSSL_INCLUDE_DIR OPENSSL_LIB_DIR OPENSSL_STATIC`.
If you add new `RUN xx-cargo` lines, copy the same `unset` prefix.

### Cache hit rate looks wrong

Inspect the registry cache manifest:

```powershell
docker buildx imagetools inspect hivehub/vectorizer-cache:buildx
```

If the cache is unexpectedly empty or only carries one arch, the
seeding `build-push.ps1` run was probably interrupted before the
cache push completed. Re-run with `-NoCache` to do a clean cold
build, then drop `-NoCache` from the second invocation to seed the
cache.

### arm64 build leaks QEMU traces in CI logs

Symptom: `grep -i qemu` against the `publish-docker (arm64)` matrix
job log returns matches.

Root cause: workflow not actually running on `ubuntu-24.04-arm` (e.g.,
matrix typo, runner unavailable).

Fix: verify the matrix `runner` field; arm64 must point at
`ubuntu-24.04-arm`, not `ubuntu-latest`.
