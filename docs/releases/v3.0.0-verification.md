## Findings

### F0 â€” `workspace.yml` schema drift (cosmetic)

On-disk `workspace.yml` at the repo root uses top-level `workspaces:`
(plural array of `{id, path, collection_name, ...}` entries left over
from a test run on 2026-04-19), but `vectorizer_core::workspace_loader`
expects top-level `workspace:`. Loader logs `WARN Failed to load
workspace configuration: YAML error: missing field 'workspace'` and
falls back to empty workspace.

- Severity: LOW (cosmetic; no runtime breakage)
- Action: cleanup task to replace `workspace.yml` with a valid shape
  (or delete if not needed for the release)

### F1 â€” `POST /batch_insert` and `POST /insert_texts` are placeholders (HIGH)

`crates/vectorizer-server/src/server/rest_handlers/vectors.rs:249-282`:
both handlers accept a `texts` array, log the count, and return
`{message, count}` success â€” but do not write a single vector into the
collection. The real write path is the single-document handler
`insert_text` (`insert.rs`).

- Severity: HIGH â€” user-facing data-loss bug on documented v3 endpoints
- Action: implement real batch insert (chunk+embed+store per text, plus
  the same replication + metrics + quota paths `insert_text` already
  runs). Or, if batch is intentionally removed, delete the routes +
  document the removal in CHANGELOG under BREAKING CHANGES.
- Blocks: probe 2.1 (snapshot round-trip wants 100 vectors in one
  batch) + probe 3.2 (batch REST surface) + probe 5.1 (10k insert
  stress)

### F2 â€” `POST /search` and `POST /collections/{name}/search` are placeholders (HIGH)

`crates/vectorizer-server/src/server/rest_handlers/search.rs:351-367`:
`search_vectors` accepts a `vector` array, returns
`{results: [], query_vector, limit}`. Both `/search` and
`/collections/{name}/search` route to this handler
(`server/core/routing.rs:257-261`). The real text-based search is
`POST /collections/{name}/search/text` (`search_vectors_by_text`,
search.rs:37), which does work.

- Severity: HIGH â€” documented search endpoint returns empty
- Action: either implement real vector search at `/search` and
  `/collections/{name}/search` (reuse `search_vectors_by_text` logic
  but with a raw `vector` input), or delete those two routes so clients
  don't get silent empty results
- Blocks: probe 2.1 (needs a working search call) + probe 2.7 (query
  cache probe relies on search hits) + probe 3.2 (batch search)

### F3 â€” `POST /collections/{name}/force-save` is a no-op (MEDIUM)

`crates/vectorizer-server/src/server/rest_handlers/collections.rs:420-450`
calls `state.store.force_save_all()`, which
(`crates/vectorizer/src/db/vector_store/autosave.rs:49-75`) just
clears the `pending_saves` set and logs "âœ… Force save completed"
without writing `.vecdb`. The real flush is owned by `AutoSaveManager`
(`crates/vectorizer/src/db/auto_save.rs:278`), which runs on a 5-minute
interval. Confirmed empirically: after a successful `/force-save`
response, `%APPDATA%\vectorizer\data\vectorizer.vecdb` does not exist.

- Severity: MEDIUM â€” endpoint returns false success; data only persists
  via the 5-min interval or graceful shutdown
- Action: make `force_save_collection` call
  `AutoSaveManager::force_save` (or expose a public `VectorStore`
  method that does so). Alternatively, delete the endpoint if the
  5-min interval is the contract.
- Blocks: probe 2.1 (needs deterministic flush between seed and
  restart)

### F5 â€” /embed, /batch_search, /batch_update, /batch_delete were placeholders (RESOLVED 2026-04-20)

While setting up probe 2.2 (real embedding model), four additional REST
handlers were discovered as hard-coded placeholders:

- `vectors.rs::embed_text` returned `vec![0.1; 512]` regardless of input.
- `search.rs::batch_search_vectors` returned `results: []`.
- `search.rs::batch_update_vectors` returned success without writing.
- `search.rs::batch_delete_vectors` returned success without deleting.

**Fix**: all four rewritten to drive the real pipeline. `/embed` now
calls `state.embedding_manager.embed(text)` and returns the resulting
vector + its real dimension. The three batch handlers share the same
`{collection, <items>: [...]}` shape and capture per-item failures
(`status: "ok"|"error"`) without aborting the batch. `batch_search`
supports both text (`query`) and raw (`vector`) queries per entry.
Initial implementation of `batch_update` deadlocked because the
collection read handle was held across the `state.store.update` write
call; fixed by capturing the dimension up front and dropping the read
handle before each write.

**Verified live**: 5 new integration tests at
`crates/vectorizer/tests/api/rest/batch_ops_real.rs` exercise the
happy path + partial-failure + empty-array rejection for each
endpoint. All pass against the release binary.

### F4 â€” default embedder emits uniform vectors (RESOLVED 2026-04-20)

**Observed symptom (probe 2.1)**: all top-5 results for "probe
document number forty two" returned `score: 1.0` with stored vectors
whose first 40+ components were ~0.0436 across every component.

**Root cause**: `Bm25Embedding::fallback_hash_embedding`
(`crates/vectorizer/src/embedding/providers/bm25.rs:245-265`)
misimplemented its pseudo-random generator. The formula was:

```rust
let value = (seed.wrapping_mul(1103515245).wrapping_add(12345 + i as u64)) % 65536;
```

For every dimension `i`, `seed * 1103515245` is a fixed constant and
only `12345 + i` varies. After `% 65536`, successive values differ by
1 (or 0 when wrapping). L2-normalizing 512 consecutive integers
produces `~1/sqrt(512) â‰ˆ 0.0442` on every component â€” exactly the
0.0436 pattern we saw.

The fallback path fires whenever the BM25 vocabulary is empty (fresh
server with no indexed corpus) â€” which is the default state for the
v3 runtime-verification probes and for first-time installs. Real
`.vecdb` corpora with a built vocabulary took the BM25 scoring path
and were unaffected.

**Fix (landed in phase8_investigate-uniform-embeddings)**: iterate
the LCG state per dimension using the PCG-family constants
`6364136223846793005` / `1442695040888963407` and extract bits
33..49 via `(state >> 33) & 0xFFFF`. Each component is now an
independent step of a proper generator seeded by the text hash.

**Verified live** (rebuilt release binary, 5 distinct texts
round-tripped through `/batch_insert` + `/collections/{name}/vectors`):

- Per-dim values now span `[-0.07, +0.07]` with both signs (was a
  constant ~0.0436).
- Pairwise cosine similarity between the 5 embeddings is in
  `[-0.06, +0.08]` (was ~1.0 â€” every vector looked identical).
- No regression in the built-vocabulary BM25 path (covered by the
  existing `embedding::providers::bm25::tests` suite).

Not a dep-bump regression â€” the bug predates the v3 migrations.
fastembed 5.4â†’5.13, ort rc.10â†’rc.11, candle 0.9â†’0.10, hf-hub 0.4â†’0.5,
and tantivy 0.25â†’0.26 are all uninvolved. Bisect ruled out.
