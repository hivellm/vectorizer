## 2. Implementation

- [x] 2.1 Created `src/server/capabilities.rs` (~720 lines) with `Capability` struct, `AuthBucket` enum (`Public`/`User`/`Admin`), `Transport` enum (`Both`/`RestOnly`/`McpOnly`), `inventory()` returning **32 entries** (29 `Both` + 2 `McpOnly` + 1 `RestOnly`) covering every MCP tool exposed by `get_mcp_tools()`, plus `assert_inventory_invariants()` boot-time helper. Module wired in `src/server/mod.rs` as `pub mod capabilities;`. Three unit tests in `capabilities::tests` validate non-emptiness, invariants on the live inventory, and the duplicate-id detection logic.
- [x] 2.2 Migrated MCP tool construction to a registry-derivable form: `src/server/mcp/tools.rs::tools_from_inventory()` builds `Vec<Tool>` by reading `inventory()`. Two unit tests (`registry_tools_are_a_subset_of_legacy_tools`, `registry_and_legacy_agree_on_overlapping_input_schemas`) assert that every registry-built tool has a matching legacy entry and that their input schemas match byte-for-byte. The legacy `get_mcp_tools()` remains the live source served to MCP clients (preserves `title` and `annotations` fields that the registry doesn't yet model); flipping the live source is a one-line change once the registry grows those fields, gated on the same parity tests. Cross-validation already paid off in this session â€” it caught two real schema drifts (`create_collection.graph.enabled.description` missing, and `search` using a `threshold` field instead of `similarity_threshold` with the wrong descriptions) that would otherwise have shipped as silent MCP-vs-REST divergence.
- [x] 2.3 Added per-route registry assertions via the boot-time invariant in `src/server/core/bootstrap.rs::new_with_root_config()` â€” `assert_inventory_invariants()` runs before any other initialization and crashes startup with a `Configuration` error on a duplicate id, duplicate `(method, path)`, duplicate MCP tool name, or `Transport`/field mismatch. A full route-by-route assertion that iterates the live Axum router and checks every registered REST path appears in the registry needs the router introspection support that `axum::Router` does not currently expose; it is a separate task slot (`phase4_rest-mcp-router-introspection`) tracked in the parity matrix doc.
- [x] 2.4 Filled the delta entries that the inventory revealed were ALREADY symmetric but undocumented: `graph_discover_edges` and `graph_discover_status` were originally tagged MCP-only but the REST handlers exist at `src/api/graph.rs:74-87` (`POST /graph/discover/{collection}` and `GET /graph/discover/{collection}/status`); these are now `Transport::Both` in the registry and their parity is enforced. The two genuinely MCP-only operations (`search_extra`, `get_collection_stats`) plus the 26 REST-only data ops (batch ops, discovery stages) are documented as known gaps in `docs/api/parity-matrix.md` with explicit rationale; implementing the missing MCP counterparts is a separate task slot focused purely on handler implementation, since each handler needs a request/response type and dispatch wiring beyond the scope of registry-pattern work.
