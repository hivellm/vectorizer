# Proposal: phase3_split-oversized-files-audit

## Why

An audit of files > 1500 lines in the repository (excluding `@generated by prost-build` gRPC files and the vendored `qdrant/` submodule) found **14 oversized scripts** that block reviewability, hide bugs, and slow IDE navigation. Three of them already have dedicated split tasks in phase3 (`split-rest-handlers-monolith`, `split-server-mod-monolith`, `split-vector-store-monolith`). The remaining **11 files** have no tracking task.

This task serves two purposes:

1. **Record the authoritative audit** (exact line counts, structural seams, split proposals) so reviewers and follow-up tasks share one source of truth.
2. **Create individual follow-up split tasks** for each uncovered file before archiving, so every oversized file is either actively being split or has an owned task waiting.

Why now: God-files amplify every other refactor cost. `phase3_reduce-unwrap-in-handlers`, `phase3_unify-error-enums`, `phase4_rest-mcp-parity-tests`, and `phase5_wire-cache-into-read-paths` all touch the same monoliths. Splitting first makes each of those tasks dramatically cheaper.

## What Changes

This task is a **tracker/audit**, not a code-split task. Its deliverables are:

1. `docs/refactoring/oversized-files-audit.md` — the full audit table (file, lines, severity, structural seams, proposed module layout) including all 14 files. Referenced from the 3 existing phase3 split tasks so they share a common plan.
2. Cross-links added to the 3 existing `phase3_split-*-monolith` proposals pointing back to this audit.
3. **New rulebook tasks created** (via `rulebook_task_create`) for each of the 11 uncovered files, each with its own proposal.md/tasks.md/specs:
   - `phase3_split-collection-monolith` (`src/db/collection.rs`, 2649)
   - `phase3_split-qdrant-grpc` (`src/grpc/qdrant_grpc.rs`, 2109)
   - `phase3_split-auth-handlers` (`src/server/auth_handlers.rs`, 1749)
   - `phase4_split-embedding-providers` (`src/embedding/mod.rs`, 1724)
   - `phase4_split-graphql-schema` (`src/api/graphql/schema.rs`, 1698)
   - `phase4_split-advanced-search` (`src/search/advanced_search.rs`, 1508)
   - `phase4_split-qdrant-api-integration-tests` (`tests/integration/qdrant_api.rs`, 1595)
   - `phase4_split-sdk-python-client` (`sdks/python/client.py`, 2907)
   - `phase4_split-sdk-javascript-client` (`sdks/javascript/src/client.js`, 2002)
   - `phase4_split-sdk-rust-client` (`sdks/rust/src/client.rs`, 1989)
   - `phase4_split-sdk-typescript-client` (`sdks/typescript/src/client.ts`, 1879)

Ordering rationale: phase3 covers core-engine monoliths (unblocks other phase3/4 work). phase4 covers providers, GraphQL, search tuning, and SDKs (can parallelize once core splits land).

4. Anti-pattern captured in knowledge base: **"Files > 1500 lines without impl-level boundaries"** with reference to this audit.

No production code is edited by this task — only documentation, memory/knowledge, and new task scaffolding.

## Impact

- Affected specs: none (meta/audit)
- Affected code: none (documentation + task scaffolding only)
- Affected docs: `docs/refactoring/oversized-files-audit.md` (new)
- Affected tasks: 3 existing `phase3_split-*-monolith` (cross-link added), 11 new tasks created
- Breaking change: NO
- User benefit: every oversized file is either under active refactor or has a tracked task; all split tasks share one canonical audit; reviewers can evaluate scope in one place.
