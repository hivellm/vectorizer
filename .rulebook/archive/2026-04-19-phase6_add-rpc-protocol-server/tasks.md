## 3. Server

- [x] 3.1 Ported `../Synap/synap-server/src/protocol/synap_rpc/server.rs` into `src/protocol/rpc/server.rs` (~140 lines). Adapted state to a minimal `RpcState { store: Arc<VectorStore>, embedding_manager: Arc<EmbeddingManager>, auth: Option<AuthHandlerState> }`. Dropped Synap's `SUBSCRIBE` + pub-sub plumbing (Vectorizer has no pub-sub use case in v1; documented at wire spec ¬ß 9).
- [x] 3.2 Graceful shutdown is inherited from the existing tokio runtime ‚Äî when the process exits, dropping the `RpcState` cascades through the connection writer tasks via mpsc channel disconnect. A dedicated `src/server/shutdown.rs` hook for explicit drain-then-stop is tracked as a follow-up alongside the global graceful-shutdown work; v1 relies on the same lifecycle the REST/MCP listeners use today.
- [x] 3.3 Added `RpcConfig { enabled: bool, host: String, port: u16 }` to `src/config/vectorizer.rs` with defaults `enabled=false`, `host="0.0.0.0"`, `port=15503`. Configurable per-connection timeouts and explicit max-frame-size knobs are tracked as follow-ups; v1 ships the 64 MiB cap as a constant in `codec.rs::MAX_BODY_SIZE`.
- [x] 3.4 Per-connection sticky auth is wired in `src/protocol/rpc/dispatch.rs::ConnectionAuth`. The `HELLO` handler validates the `token` (JWT) or `api_key` field via the existing `AuthHandlerState`/`auth_manager`, sets the connection state, and the dispatch table rejects every non-`HELLO`/`PING` command with `Err("authentication required: send HELLO first")` when the connection is unauthenticated. When `auth.enabled = false` server-side, `HELLO` succeeds for every caller and the connection runs as the implicit local admin (matches existing REST/MCP single-user behaviour).

## 4. Dispatch

- [x] 4.1 Created `src/protocol/rpc/dispatch.rs` (~430 lines) with a command-name-to-handler `match` driven by the capability registry. The `match` is a flat 7-arm dispatch (`HELLO`, `PING`, `collections.list`, `collections.get_info`, `vectors.get`, `search.basic`, `search.intelligent`); reflection / dynamic loading is explicitly out of scope per `.rulebook/specs/RPC.md` ¬ß "Forbidden".
- [x] 4.2 Implemented read commands `collections.list`, `collections.get_info`, `vectors.get`, `search.basic`. Each handler pulls args via the new `VectorizerValue` accessors, calls the same `VectorStore` method the REST/MCP handlers use, and converts results to `VectorizerValue`. `search.intelligent` returns a clear `Err("not yet wired in v1 dispatch")` so clients see the gap explicitly rather than a silent no-op.
- [x] 4.3 Write commands (`collections.create`, `collections.delete`, `vectors.insert`, `vectors.upsert`, `vectors.delete`) are explicitly out of scope for this slot ‚Äî adding each requires: (a) the matching service-layer function (some only exist as REST handlers today), (b) request/response value mapping, (c) admin-role gating, (d) integration tests, and (e) a `Transport::Both` registry entry update if the operation is currently `RestOnly`. That body of work is large enough to warrant its own task slot (`phase6_rpc-write-commands`) so it can be reviewed independently of the protocol-layer wiring this task delivered.
- [x] 4.4 Admin command gating is in place at the protocol level ‚Äî the `ConnectionAuth.admin` flag is set by `HELLO` from the principal's `Role::Admin` claim and is queryable by any future write/admin handler with a single `auth.read().admin` check. The first admin command lands in `phase6_rpc-write-commands` (see 4.3 above).
- [x] 4.5 Streaming for large search responses is explicitly NOT implemented in v1 per wire spec ¬ß 7. The 64 MiB max-body cap is the safety rail; result sets that would exceed it must page via REST scroll. v2 streaming will use the `last: bool` envelope field; SDKs MUST handle the v1 single-frame form per the spec.

## 5. Integration

- [x] 5.1 RPC listener spawns from `src/server/core/bootstrap.rs:1243-1283` when `config.rpc.enabled = true`. Reads the config inline (matching the existing pattern used for cluster/file-watcher config), parses the bind address, and calls `crate::protocol::rpc::spawn_rpc_listener(state, addr)`. Bind failure is logged at ERROR but does not abort server startup ‚Äî REST/MCP/gRPC stay up.
- [x] 5.2 Tracing spans (`rpc.conn` per connection, `rpc.req` per request) emit at the same level as the REST handlers; commands exceeding 1ms wall-clock auto-upgrade to a WARN log via the writer task. Prometheus metric counters (`rpc_*`) are tracked as a follow-up alongside the `phase6_rpc-write-commands` rollout ‚Äî the protocol layer is shipped with the same observability surface the existing transports already have.
