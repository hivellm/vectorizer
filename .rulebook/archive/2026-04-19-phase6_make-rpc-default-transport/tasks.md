## 3. Deployment manifests

## 4. CLI + SDK defaults

- [x] 4.1 The original item targeted `src/bin/vectorizer-cli.rs` as if it were a client CLI; verified it is not. `vectorizer-cli` is the **server-management** binary (subcommands `Start`, `Stop`, `Status`, `Install`, `Uninstall`, `Setup`) — it boots and supervises the server, it does not connect to a remote one. There is no "default client transport" to flip in this binary. The actual default-transport handover lives in each SDK's constructor (per the `parseEndpoint` items added to every `phase6_sdk-*-rpc/tasks.md` in commit `c5dc41f6`). When `vectorizer-cli` gains a future `query` / `search` subcommand that talks to a remote server, it will reuse the canonical URL parser from the relevant SDK and inherit the `vectorizer://` default automatically — no extra coordination needed.
- [x] 4.2 SDK README quickstarts are updated by each `phase6_sdk-{lang}-rpc` task at the moment that task ships its RPC client (proposal "Default URL scheme" sections + tasks.md `parseEndpoint` items, all added in commit `c5dc41f6`). Updating those READMEs from this task slot before the SDK code lands would point users at non-existent code; honouring the dependency boundary is the correct action.
