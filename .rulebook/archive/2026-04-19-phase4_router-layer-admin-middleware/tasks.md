## 1. State unification

- [x] 1.1 The combined `RouterState` was unnecessary because the actual blocker for the original router-level attempt was a hidden `Send` violation in `extract_auth_from_request(&state, &request).await`, not a state-type unification problem. Once `extract_auth_from_parts(state, &HeaderMap, Option<&str>)` was introduced (both borrowed types are `Send + Sync`) the typed `from_fn_with_state` form composes cleanly across the existing two state types via `with_state` erasure to `Router<()>` followed by `merge`. Capturing the rationale in `docs/api/route-auth-matrix.md` § "Why this works now (and didn't before)".
- [x] 1.2 Handler signatures unchanged for the 5 admin handlers we touched — the per-handler `_admin: AdminAuth` extractor is removed, no other parameter shifts. The 4 `auth_handlers::*` admin handlers (`/auth/users*`) still keep their `State<AuthHandlerState>` + inline admin check; lifting them onto the same `admin_router` is gated on a state-type unification documented as a follow-up task `phase4_unify-admin-auth-handlers-state` in the route-auth matrix.

## 2. Router split

- [x] 2.1 Built `admin_router: Router<()>` in `src/server/core/routing.rs:740-768` with all 9 admin POST routes (`/workspace/{add,remove,config}`, `/setup/{apply,browse}`, `/config`, `/admin/restart`, `/backups/{create,restore}`). State injected via `.with_state(self.clone())` at build time.
- [x] 2.2 Applied `axum::middleware::from_fn_with_state(auth_state, require_admin_middleware)` on `admin_router` when `auth_handler_state` is `Some`. When auth is globally disabled (single-user mode), the layer is omitted to preserve legacy behaviour where every caller is treated as admin.
- [x] 2.3 The existing `auth_middleware` was not re-applied on a dedicated authenticated router because the production-mode global middleware in `routing.rs:761` already populates `Extension<AuthState>` for every non-public request. Adding a duplicate would double-validate every JWT — the `AdminAuth` extractor and `require_admin_middleware` both run their own validation today and would still need the `Extension` populated by the global middleware in non-production mode. Migrating the global middleware to a typed `auth_middleware` layer is tracked separately as a refactor of the inline closure at `routing.rs:761-871`.
